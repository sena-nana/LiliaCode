# LiliaCode 视觉设计标准

## 0. 适用范围

本文定义 LiliaCode 的视觉语言，用于指导桌面端界面、文档截图、后续功能界面和 UI 改动评审。

LiliaCode 是面向代码工程的 Agent 工作台。它不是营销站点、通用聊天应用或装饰性仪表盘。界面应帮助开发者理解项目上下文、任务状态、Agent 过程、权限边界和待处理的人机决策。

## 1. 设计原则

### 1.1 产品气质

- 克制、清晰、可扫描。
- 信息密度适合工程工作，但不制造噪音。
- 默认保持安静，只在用户必须行动时提高可见度。
- 能呈现过程，但不让过程日志抢走视觉中心。

### 1.2 视觉优先级

1. 主内容：当前任务、时间线、输入工作区、主操作。
2. 当前状态：运行中、等待中、选中态、权限、计划模式、连接状态。
3. 过程信息：工具、文件、命令、Todo、队列、上下文提示。
4. 辅助操作：设置、插件、排序、折叠、二级管理入口。

每个主界面都应让用户快速回答：

- 我现在在哪里？
- Agent 正在做什么？
- 下一步是否需要我决策？

## 2. 布局系统

### 2.1 应用壳层

- 标题栏：位置面包屑、侧栏开关、对话侧栏开关、窗口控制。
- 左侧栏：项目、收集箱、会话树、搜索、连接状态。
- 主工作区：当前页面内容。
- 聊天任务界面：时间线是主要阅读区域，Composer 是主要行动区域。
- 管理页面：设置、插件、项目和相关配置。

### 2.2 信息密度

- 优先使用紧凑行、稳定控件和可预期的扫描路径。
- 避免落地页式构图、hero 区块、超大标题、装饰面板和营销式卡片流。
- 避免嵌套卡片。卡片只用于独立重复项、弹窗和工具容器。
- 侧栏行、图标按钮、chip、tab、时间线节点和 Composer 控件需要保持稳定尺寸。

### 2.3 空间规则

- 壳层和侧栏元素应像桌面框架的一部分，而不是漂浮装饰。
- 主内容应保留足够阅读空间，但避免空洞的仪式化留白。
- 处理 Agent 交互的控件应靠近 Composer 或待处理区域，不只藏在时间线深处。

## 3. 色彩令牌

优先使用 `apps/desktop/src/styles/` 中已有 CSS 变量和标准组件类。新增界面应先从这些令牌与 `.ui-*` 标准类派生，再考虑局部新增颜色或样式扩展。

### 3.1 表面

| 令牌 | 用途 |
| --- | --- |
| `--bg` | 主工作区表面。 |
| `--bg-elev` | 标题栏、侧栏、卡片、浮层、Composer 外壳。 |
| `--bg-subtle` | 输入框、代码块、chip、非激活列表表面。 |
| `--bg-hover` | 行和轻量控件的 hover 背景。 |
| `--bg-active` | 激活行、选中 tab、按下状态。 |

### 3.2 边线

| 令牌 | 用途 |
| --- | --- |
| `--border-soft` | 低强调分隔线和内部边界。 |
| `--border` | 标准组件边界。 |
| `--border-strong` | 浮层、激活输入、强交互边界。 |

### 3.3 文本

| 令牌 | 用途 |
| --- | --- |
| `--text` | 主文本和可读内容。 |
| `--text-muted` | 次要标签、元信息、非激活控件。 |
| `--text-faint` | 占位、路径、弱提示、空状态。 |

### 3.4 强调色与语义色

| 令牌 | 用途 |
| --- | --- |
| `--accent` | 当前项、主操作、焦点、轻量激活状态。 |
| `--accent-strong` | 必要时使用的更强强调交互。 |
| `--accent-soft` | 选中 / 激活的柔和背景、轻量正向状态。 |
| `--ok` / `--ok-soft` | 成功和健康状态。默认保持低强调。 |
| `--warn` / `--warn-soft` | 警告、部分配置、大上下文、潜在风险。 |
| `--err` / `--err-soft` | 错误、危险操作、失败、打断。 |

### 3.5 色彩规则

- 用中性灰阶建立结构。
- 蓝色只用于当前、主操作和焦点，不做大面积装饰。
- 语义色必须表达状态或风险。
- 危险动作通常使用柔和红色背景加红色文本。
- 深浅主题都必须通过共享变量保持可读。

## 4. 排版

### 4.1 字体

- 界面文本：`--font-sans`。
- 代码、命令、路径、日志：`--font-mono`。

### 4.2 字号

| 字号 | 用途 |
| --- | --- |
| 18px | 页面标题或管理页主标题。 |
| 14px | 默认正文和主要工作文本。 |
| 13px | 行标签、工具栏标签、紧凑标题、控件文本。 |
| 12px | 元信息、提示、徽章、次级说明。 |
| 10-11px | 数量、微型徽章、来源标签、低强调工具文本。 |

规则：

- 字号不随视口宽度缩放。
- 字距保持 `0`，除非已有组件确实需要其他设置。
- 中文界面文案保持短、直、可操作。按钮写动作，状态写事实。

## 5. 组件语言

### 5.1 按钮

- 熟悉工具优先使用图标按钮：折叠、搜索、附件、计划、重试、删除、设置、窗口控制。
- 有可用图标时优先使用 `lucide-vue-next`。
- 需要明确语义的命令使用文字按钮。
- 主发送动作为紧凑圆形强调按钮；打断动作使用错误色。

### 5.2 列表与行

- 行需要支持 hover、active、muted 和 disabled 状态。
- 当前行使用激活背景加强调色文本。
- 次级行或子行可以使用弱化文本，但必须保持可读。
- 树拖拽 / 放置状态应提供即时但克制的反馈。

### 5.3 Chip、徽章与 Pill

- Chip 用于紧凑选项、附件、权限和下拉触发。
- Pill / 徽章用于可读状态和数量。
- 药丸形适用于徽章、chip、状态点和紧凑发送类控件。

### 5.4 卡片与面板

- 卡片用于独立信息组、重复项、弹窗和被框定的工具。
- 不把页面 section 做成装饰性卡片。
- 不嵌套卡片，除非内部元素是真正的重复项。

### 5.5 输入

- 输入使用柔和背景、清晰边线和可见焦点。
- Composer 输入区是工作区，不是普通 textarea。
- 内联文件引用、粘贴图片、粘贴路径和长文本应共享一套上下文模型。

### 5.6 菜单与浮层

- 使用提升表面、强边线和 6-8px 圆角。
- 菜单应紧凑，并支持键盘可读。
- 危险菜单项使用错误文本和柔和红色的 pending / confirm 状态。

## 6. Agent 时间线

时间线是核心视觉对象。它展示过程，但不抢走用户消息和最终回复的主线地位。

- 用户消息和最终回复保留在主对话线上。
- 工具调用、命令、文件变更、计划、Todo 和调试事件作为过程节点。
- 已完成的过程节点应回落为低强调摘要。
- 运行中和等待中的状态可以更显眼。
- 需要用户处理的决策也必须在 Composer 或待处理区域附近出现。
- 时间线展示语义通过 `packages/contracts` 派生，外部工具不得注入任意 UI 组件。

## 7. Composer 与待处理交互

Composer 既是输入工作区，也是人的决策入口。

### 7.1 普通状态

- 富文本输入。
- 内联上下文引用。
- 附件预览。
- 权限模式。
- 计划模式。
- 发送 / 打断。

### 7.2 待处理状态

- AskUser、工具确认和计划确认会临时接管 Composer 区域。
- 待处理面板应说明当前正在回答什么、有哪些选项、能否跳过、拒绝、编辑或要求修改。
- 计划正文属于时间线里的计划事件；Composer 待处理 UI 不重复展示长计划内容。

### 7.3 运行状态

- 运行中且 Composer 为空：发送动作变为打断。
- 运行中且 Composer 非空：新内容进入队列。

## 8. 状态与反馈

### 8.1 状态层级

- 选中 / 当前：激活背景 + 强调色文本。
- Hover：轻量提升背景。
- Disabled：弱文本，不参与视觉竞争。
- Success：默认低强调，除非结果必须被注意。
- Warning：使用警告色并说明影响。
- Error / 危险：使用错误色并说明后果。

### 8.2 焦点

- 键盘焦点必须可见。
- 优先使用强调色边框或 outline，并确保深浅主题都有足够对比。

### 8.3 动效

- 动效用于表达连续性：侧栏折叠、面包屑切换、待处理面板过渡、搜索浮层、树展开。
- 过渡保持短促、功能性。
- 不实现会关闭 Lilia 既有动效的 `prefers-reduced-motion` 行为。

## 9. 文案

界面文案应像工程工具：直接、短、准确。

推荐模式：

- `正在加载对话…`
- `加入调度队列`
- `要求修改计划`
- `完全访问`
- `询问`
- `只读`

规则：

- 按钮写动作。
- 状态文本写事实。
- 提示写影响。
- 错误文本说明失败对象，并在可能时给出下一步。
- 避免营销口号、情绪化鼓励，以及在应用内解释视觉设计意图。

## 10. 评审清单

发布视觉或交互改动前，检查：

- 界面仍然像克制的工程工作台。
- 主内容强于状态和辅助控件。
- 新颜色来自现有令牌。
- 深浅主题都可读。
- 文本不会溢出按钮、chip、行、卡片或面板。
- 控件在 hover、active、loading、empty 状态下尺寸稳定。
- 待处理 Agent 交互在 Composer 附近可见。
- 卡片没有嵌套，也没有被用作装饰性页面 section。
- 图标沿用现有图标语言。
- 如果引入新的交互模式，先确认是否需要 IAB 评审。
