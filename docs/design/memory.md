# Memory System 设计文档

> **状态:v1 规划中,尚未实现。** 本文记录已确定的设计方向、未决问题与待办事项,作为后续实施的单一信源。
> README 中 Memory 小节是对外摘要,**实现细节以本文为准**。

---

## 1. 设计动机

Memory 的目标是让 Lilia **跨会话**沉淀知识——用户偏好、项目决定、历史结论。

要回避的反模式:

- **全量前缀拼接**(`<memory scope="...">` 一股脑塞进 prompt):token 一过 30k 就开始触发 Lost-in-the-Middle,记忆越多反而越稀释注意力
- **让 agent 自己决定何时查**:LLM 默认偏向不调用可选工具(Toolformer / MemGPT 后续研究一致结论),Memory 会沦为"存了但没人查"

设计要在这两个失败模式之间找平衡:**inference-time speculative retrieval** —— 把"是否检索"的决策从 agent 身上拿走,交给一个旁路的外置模型;检索结果不直接污染 agent 上下文,而是引导 agent 自己决定要不要拉取。

---

## 2. 数据层

### 2.1 作用域

只有**用户级**和**项目级**两层,刻意不做更细粒度(没有 task 级 / session 级)。

允许从草稿对话保存(`sourceTaskId: "draft:<id>"`),因为记忆的载体可以游离于结构化任务之外,记忆比会话长寿。

### 2.2 VCC 对话链

VCC 是把对话整理为"对话链"的存储算法,关键性质:

- **基本不做压缩**——保留原文,避免摘要丢失信息
- **chunking 边界对齐到一次完整对话**(user ↔ assistant 一来一回),保证 grep 命中的最小单元是语义完整的对话单元,不会出现"抓到半句话、缺失前后文"的经典 RAG 断章问题
- 解决的是 chunking strategy 问题,**不解决语义召回盲区**(关键词改写 / 同义词跨越仍然需要外置模型补)

---

## 3. 检索层

### 3.1 外置模型并行观察

一个**低成本外置模型**(预计 Haiku 4.5 或更小)持续观察 agent 当前的执行思路:

- 输入:agent 的 thinking blocks + tool calls + 最近 N 轮对话
- 输出:决定是否触发检索、检索什么关键词、命中后是否回送引导消息
- 与 agent 推理**并行执行,互不阻塞**——避免 agent 等检索

### 3.2 Grep over 对话链

外置模型的查询通过 grep 命中 VCC 对话链:

- 优点:可解释、零幻觉、确定性、复用 Claude Code 已成熟的工具调用范式
- 局限:关键词匹配,对中英混合、同义改写、缩写召回率差
- 兜底:外置模型应在 grep 之前先做查询扩展(同义词 / 多语言变体)

## 4. 注入层

### 4.1 命中后的处理

检索命中后**不直接塞进 agent 上下文**,而是把"有相关记忆"的引导消息回送到 agent 对话,由 agent 自行决定是否通过 grep 拉取完整内容。

### 4.2 三层时机策略

**TODO Layer 1 — 基线注入(必做,v1)**

session 启动时把「用户级强偏好 + 项目级硬约束」作为 system prompt 的一部分一次性注入。

- 入库标准从严,只放**真正跨任务通用的约束性条目**(例如「PR 描述里不要出现 emoji」「DB 迁移必须先走 dry-run」),不放参考信息
- 这不算"事中介入",是 RAG 基线
- 风险:选不准会退化为"全部记忆拼前缀"那种污染——所以入库标准是 v1 的关键校准点

**TODO Layer 2 — 机会窗口(主战场,v1)**

拦截 tool result 返回的 hook 点,外置模型 sub-second 判断后,把引导消息追加到下一轮 user / system message。

- 利用 agent 完成工具调用后的**自然状态转换点**——刚获得新信息、即将形成新假设——把打断成本降到最低
- 大多数"应该查记忆"的场景,agent 是在看到工具结果后才意识到需要的
- 工程约束:外置模型决策必须在 1 秒内返回,否则错过窗口
- 局限:纯 reasoning、无 tool call 的短任务拿不到这个窗口,但这种任务通常也不需要记忆助理

**TODO Layer 3 — 救火式(延后到 v2)**

监测 agent 反复试错 / thinking 出现犹豫词 / 同一工具调用超阈值时触发。

- 此时 agent 自己也意识到当前路径有问题,引导消息**更易被接受而非被当作噪声**
- 但"检测卡住"本身是个难题,误判率高
- 触发阈值要高(宁可不救,不能误救)
- **v1 不做**,先把 Layer 1+2 跑稳,根据真实采纳率数据再决定是否上 Layer 3

### 4.3 跨层兜底

无论选哪层,以下都是"不做早晚翻车"的基础设施:

- **TODO** Session 级冷却期:单 session 内引导消息频率上限(默认每 5 turn 至多 1 次),防止外置模型疯狂打断
- **TODO** 引导消息使用可识别前缀(如 `[Lilia 记忆助理]`),让模型识别这是辅助提示而非用户新指令——framing 直接决定 agent 把它当多重要,研究里 framing 的影响经常超过内容本身
- **TODO** 前端 emergency stop:用户可一键关闭记忆助理。没有这个用户不敢把它常开
- **TODO** 采纳率埋点:引导消息发出后 N 轮内被 agent 实际引用的比例。**这是判断 Layer 2/3 是否值得存在的唯一硬指标**,长期 < 20% 应提高外置模型触发阈值

## 5. 与前置任务上下文的关系

前置任务(`dependsOn`)的上下文注入走**同构思路**,但策略有差异:

- 前置任务是**用户显式声明的强相关信号**,不应该让 agent 自决"要不要查前置任务"——它经常会判断错
- **TODO** 默认拉取前置任务的最终摘要塞进新 session 的初始 system prompt,agent 仍能用工具拉完整对话,但不需要自己判断"要不要看"
- Memory 走"agent 自决",前置任务走"默认注入摘要"——同构是设计洁癖,分治更符合实际信号强度

## 6. 已知风险

| 风险 | 性质 | 缓解 |
|---|---|---|
| Reasoning 链被打断 | agent 收到引导后丢失原本思路、反复横跳 | Layer 2 选在 tool result 后这种自然状态转换点;采纳率埋点持续监控 |
| Underuse bias | 即使外置模型代为决策,agent 仍可能把引导当噪声忽略 | 引导消息可识别前缀 + framing 优化(措辞每改一个词,调用率能差 3 倍) |
| 召回盲区 | grep 抓不到语义改写(「装填依赖」vs「安装包」) | 外置模型在 grep 之前做查询扩展;长期可加向量召回作为补充 |
| 注入身份归属难题 | system / user / tool result / 自定义角色,每种都有副作用 | 早期固定一种(倾向 user message + 前缀),不要让这个变量随机变化 |
| 外置模型自身成本 | 频率高会累积可观 token 消耗 | 触发节流 + 信号弱时 short-circuit 不触发,让 agent 完全无感 |
